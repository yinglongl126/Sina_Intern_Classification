# 商品/服务实体抽取设计

## 1. 为什么实体抽取是第一核心模块

### 1.1 问题背景

当前数据（`电商平台25年12月投诉数据.xlsx`，24713条）中不包含直接的商品名称或商品品类字段。商品/服务信息分散在以下字段中：

| 字段 | 是否包含商品信息 | 示例 |
|------|----------------|------|
| 标题 | ★★★★★ 最核心 | "买了两万多的显卡，温度过高一直黑屏不能使用" |
| 内容 | ★★★★ 重要补充 | 正文中详细描述商品细节 |
| 商家 | ★★★ 间接线索 | "XX数码专营店" → 推测为数码品类 |
| 诉求 | ★ 偶尔有用 | "退货退款" → 暗示有实物商品 |
| 链接 | ★★ 潜在信息源 | 商品URL可能包含类目路径 |

**核心结论：不能跳过实体抽取直接做分类。必须先抽取出"这个投诉在说什么商品/服务"，才能谈分类。**

### 1.2 实体抽取在系统中的位置

```
投诉数据进入
    │
    ▼
┌─────────────────────────────────┐
│ Step 1: 文本预处理               │
│ - 字段提取、清洗、融合           │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ Step 2: 商品/服务实体抽取 ← 本模块 │
│ - 商品词、品牌词、服务词、型号   │
│ - 确定主商品/服务对象            │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ Step 3: 品类分类                 │
│ - 基于抽取的实体 + 原文分类       │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│ Step 4: 置信度评估 + 分流决策     │
└─────────────────────────────────┘
```

## 2. 抽取目标

实体抽取要回答三个问题：

1. **投诉涉及的商品或服务是什么？**（核心问题）
2. **商品/服务的具体属性是什么？**（品牌、型号、规格）
3. **存在一个还是多个商品/服务？**（主次判断）

抽取结果直接驱动后续的品类分类。

## 3. 实体类型体系

### 3.1 七类实体

| 实体类型 | 代码 | 说明 | 示例 |
|---------|------|------|------|
| 商品词 | PRODUCT | 明确的商品名称或品类词 | 显卡、手机、羽绒服、奶粉 |
| 品牌词 | BRAND | 商品或服务品牌 | 华为、Nike、帮宝适、星巴克 |
| 型号/系列词 | MODEL | 具体型号、系列代号、产品线 | RTX 5090、iPhone 15、Mate 60 |
| 服务词 | SERVICE | 明确的服务类型名称 | 会员、外卖、课程、运费险、账号 |
| 平台/渠道词 | PLATFORM | 电商平台名称 | 淘宝、拼多多、抖音商城 |
| 店铺/商家线索 | STORE | 店铺名中包含的品类或品牌线索 | "XX手机专卖店"中的"手机" |
| 属性词 | ATTRIBUTE | 商品属性/规格/材质/风格描述 | 加绒、纯棉、256G、XL、防水 |

### 3.2 实体类型不是投诉问题类型

**以下内容不是商品/服务实体，应在实体抽取阶段主动排除：**

| 不应抽取的内容 | 原因 | 示例 |
|-------------|------|------|
| 投诉问题词 | 属于投诉问题维度 | 退款、退货、质量差、虚假宣传、发货慢、客服态度差 |
| 诉求词 | 属于消费者诉求 | 退款退货、赔偿、处罚商家 |
| 处理状态词 | 属于流程维度 | 处理中、已解决、待审核 |
| 情感/主观评价词 | 属于情感分析维度 | 太差了、坑人、垃圾 |

**识别规则：** 如果无法回答"这个投诉的对象是什么"这个问题，那它就不是商品/服务实体。

## 4. 字段抽取优先级与策略

### 4.1 优先级规则

```
标题 > 内容首段 > 内容全文 > 商家名称 > 诉求 > 商品链接（仅作辅助线索，非第一阶段依赖）
```

### 4.2 分字段抽取策略

#### 4.2.1 从"标题"抽取（最高优先级）

标题通常直接包含核心商品词，抽取策略：

1. **直接匹配商品词典** — 如果标题中包含已知商品词（如"显卡"、"羽绒服"），直接抽取
2. **品牌+型号推断** — 如"5090" → 型号词；配合上下文推断为显卡型号
3. **数量词+名词模式** — 如"一双凉靴" → "凉靴"为商品词
4. **服务关键词匹配** — 如"会员"、"外卖"、"充值"

**示例：**
- 标题："买了两万多的显卡，温度过高一直黑屏不能使用"
  → 商品词: "显卡" | 金额线索: "两万多"
- 标题："拼多多购主机半年出问题"
  → 商品词: "主机" | 平台: "拼多多"
- 标题："拼多多无据退盲盒款，98元打了水漂"
  → 商品词: "盲盒" | 金额: "98元" | 平台: "拼多多"

#### 4.2.2 从"内容"抽取（补充和确认）

内容正文是标题的补充，策略：

1. **确认标题中的商品词** — 在正文中寻找相同的商品词作为确认
2. **补充细节** — 从正文中提取品牌、型号、规格等细节
3. **发现第二商品** — 正文可能提到标题未出现的次要商品
4. **解析服务场景** — 服务类投诉往往在正文中描述服务场景

**示例：**
- 标题："要求退货退款"（无商品词）
  → 从正文中搜索商品词和品牌词
- 正文："本人12月4号在淘宝购买了5090显卡...显卡温度过高"
  → 商品词: "显卡" | 型号: "5090" | 平台: "淘宝"

#### 4.2.3 从"商家"抽取（间接线索）

商家名称有时包含品类线索：

| 商家名称 | 可推断的线索 | 置信度 |
|---------|------------|--------|
| "XX数码专营店" | 数码电子类商品 | 中 |
| "XX服饰旗舰店" | 服装类商品 | 中 |
| "XX食品专营店" | 食品类商品 | 中 |
| "XX严选" | 无法推断 | 无 |
| "阿里客服" | 无法推断（这是平台客服，不是商家） | 无 |

**注意：** 
- 如果商家名称是平台客服（如"阿里客服"、"拼多多客户服务"），不能用于品类推断
- 商家线索仅作为辅助，置信度较低，需要标题和正文的交叉验证

#### 4.2.4 从"链接"抽取（辅助线索，非第一阶段依赖）

**定位：链接字段仅作为辅助线索，第一阶段不依赖链接解析。**

URL中可能包含类目路径：

- `https://item.taobao.com/item.htm?id=...` → 无类目信息
- `https://detail.tmall.com/item.htm?cat=50024452` → 可解析类目ID（但需登录/反爬）
- `https://item.jd.com/100012345.html` → 需爬取页面才可获取类目

**第一阶段限制与原则：**

1. **链接字段不可靠：**
   - 多数投诉平台的URL已经是跳转后的短链，无法直接解析
   - 淘宝/天猫/京东等电商平台的类目信息需要登录、可能触发反爬、链接可能已失效
   - 实际可从URL中获取类目信息的比例可能不到5%

2. **第一阶段不依赖链接解析：**
   - 实体抽取和品类分类**完全基于文本字段**（标题、内容、商家、诉求、问题）
   - 链接字段仅在后续阶段作为增强能力考虑
   - **如果链接解析失败，不应影响主流程的分类结果**

3. **后续阶段可探索的方向：**
   - 如果有稳定的商品ID→类目映射数据库（如事先爬取的类目字典），可作为辅助特征
   - 可尝试从投诉平台获取商品类目字段（见 Q20：是否需要接入电商平台原始商品类目）
   - 链接解析始终作为"锦上添花"，不作为必要条件

#### 4.2.5 从"诉求"和"问题"字段抽取（最低优先级，仅作排除用）

这些字段主要包含投诉问题类型，**不能直接作为商品实体来源**，但可用于：

- **排除判断：** 如果诉求是"退款退货"，说明投诉对象是实物商品（但不告诉我们是什么商品）
- **辅助确认：** 如果问题字段包含商品词（如"显卡过热"），可作为交叉验证

## 5. 多实体场景处理

### 5.1 确定主商品/服务对象

当抽取到多个商品/服务实体时，按以下规则确定主对象：

| 优先级 | 规则 | 说明 |
|--------|------|------|
| 1 | 标题首次提及 | 标题中第一个出现的商品/服务词，通常是核心对象 |
| 2 | 金额关联 | 金额描述最接近的商品对象 |
| 3 | 问题关联 | 在"问题"描述附近出现的商品词 |
| 4 | 位置权重 | 越靠前出现的商品词权重越高 |

### 5.2 多实体输出格式

```json
{
  "primary_entity": {
    "text": "显卡",
    "normalized": "NVIDIA GeForce RTX 5090 显卡",
    "type": "PRODUCT",
    "source_field": "标题",
    "confidence": 0.95
  },
  "secondary_entities": [
    {
      "text": "显示器",
      "normalized": "显示器",
      "type": "PRODUCT",
      "source_field": "内容",
      "relation_to_primary": "关联设备"
    }
  ]
}
```

### 5.3 特殊场景

**场景1：捆绑销售/套装**
- "买的护肤套装" → 主实体: 护肤套装 | 辅实体: [具体单品列表]

**场景2：买A送B**
- "买了手机送的耳机坏了" → 主实体: 手机（购买对象）| 辅实体: 耳机（赠品，但是投诉焦点）

**场景3：多件同类商品**
- "买了三件衣服都有问题" → 主实体: 衣服 | 不需要拆分为三个实体

**场景4：商品+服务混合**
- "买的手机，送的会员不能激活" → 主实体: 手机 | 辅实体: 会员服务

## 6. 无商品实体时的处理

### 6.1 无商品实体的常见情况

| 情况 | 示例投诉 | 处理方式 |
|------|---------|---------|
| 信息完全缺失 | "太差了，投诉" | → 标记 need_human_review=true, 分类归入"暂无法归类" |
| 只有投诉问题 | "退款一直不到账" | → 无法确定品类，标记为需要人工复核 |
| 投诉对象是平台整体 | "这个平台太差了" | → 归入"平台服务与账号服务"，置信度设置低 |
| 投诉对象是商家行为 | "商家辱骂消费者" | → 从商家名称推断，如无法推断则人工复核 |

### 6.2 兜底策略

```
1. 如果标题+内容都无商品/服务词 → 
   a. 检查商家名称是否含品类线索
   b. 如果商家名也无线索 → need_human_review=true
   c. 有线索 → 以低置信度分类

2. 如果只有投诉问题词（退款/虚假宣传等）→
   a. 标记 entity_extraction_failed=true
   b. 归入"其他/待人工复核" → "暂无法归类" → "信息不足"
```

## 7. 词典建设

### 7.1 商品词典

**目标：** 覆盖电商投诉中最常见的商品词及其同义词、别名。

**建设方式：**

**阶段一（手动+LLM辅助）：**
- 从24713条投诉标题中，使用大模型批量抽取商品词
- 人工审核去重，标准化商品词
- 为每个商品词标注对应的L1/L2品类
- 目标：覆盖>80%投诉中出现的商品词（~500-1000个商品词）

**阶段二（持续积累）：**
- 从人工复核的投诉中提取新的商品词
- 从电商平台类目树中导入标准商品词
- 建立同义词映射（"5090" → "显卡"）

**词典条目示例：**
```json
{
  "term": "显卡",
  "aliases": ["图形卡", "GPU", "显示适配器", "独显", "核显"],
  "category_l2": "电脑整机与硬件",
  "category_l1": "数码电子",
  "type": "PRODUCT",
  "frequency": "HIGH"
}
```

**数据来源：**
- 公开品牌数据库

**品牌条目示例：**

## 8. 品牌歧义处理

### 8.1 歧义品牌示例

| 品牌 | 可能的品类 | 歧义程度 |
|------|----------|---------|
| 苹果/Apple | 手机、电脑、平板、手表、耳机 | 高 |
| 小米/Xiaomi | 手机、电视、扫地机器人、空气炸锅、手环、充电宝 | 极高 |
| 华为/Huawei | 手机、电脑、平板、手表、电视、路由器 | 高 |
| 三星/Samsung | 手机、电视、冰箱、洗衣机、存储卡 | 高 |
| 索尼/Sony | 电视、相机、游戏主机、耳机 | 高 |

### 8.2 歧义消解策略

```
1. 检查投诉文本中的上下文关键词
   - "小米电视" → 家用电器 > 大家电
   - "小米手机" → 数码电子 > 手机通讯

2. 如果上下文无法消解歧义：
   - 列出所有可能的品类
   - 置信度打折
   - 标记 need_human_review

3. 如果品牌有明确的默认品类（如"苹果"在消费语境下默认=手机）：
   - 按默认品类分类，但置信度略降（如0.85 → 0.75）
   - 在reason中说明假设
```

## 10. 实体抽取技术方案

### 10.1 阶段一：LLM抽取（冷启动，第1-3周）

**做法：** 使用大模型Prompt进行实体抽取，与品类分类共用一次LLM调用。

**Prompt设计要点：**
- 明确要求抽取商品词、品牌词、型号、服务词
- 明确要求排除投诉问题词
- 要求输出结构化JSON
- 对无实体的投诉要求明确标记

**优点：** 无需训练数据，直接可用
**缺点：** 成本和延迟较高
**覆盖：** 冷启动阶段100%用LLM抽取

### 10.2 阶段二：词典+规则为主，LLM兜底（第3-6周）

**做法：**
1. 商品词典精确匹配 + 模糊匹配（前缀、后缀、拼音）
2. 品牌词典匹配
3. 规则：数量词+名词、品牌+型号等模式
4. LLM处理规则无法覆盖的case

**覆盖目标：** 词典+规则覆盖70%，LLM覆盖30%

### 10.3 阶段三：专用NER模型（第6周+）

**做法：**
1. 用阶段一、二积累的实体标注数据
2. 训练基于BERT的序列标注模型（BIO标注体系）
3. 识别PRODUCT、BRAND、MODEL、SERVICE、STORE、ATTRIBUTE、PLATFORM七类实体

**训练数据需求：** ~2000条有实体标注的投诉文本
**预期效果（阶段二）：** F1 > 0.85（需≥500条实体标注数据支撑，阶段一不做此承诺）

## 11. 实体抽取评估指标

### 11.1 阶段一评估方式（人工抽样评估）

阶段一不承诺自动计算的 F1 指标（标注数据尚不足）。阶段一采用人工抽样评估：

| 评估指标 | 定义 | 阶段一目标 | 计算方式 |
|---------|------|----------|---------|
| 实体命中率（Entity Hit Rate） | 至少抽到≥1个商品/服务实体的投诉占比 | ≥85% | 人工逐条检查 |
| 实体漏抽率（Entity Miss Rate） | 投诉中明确有商品/服务实体，但系统未抽取到的比例 | ≤10% | 人工抽样判定 |
| 实体误抽率（False Extraction Rate） | 将非商品/服务词（如投诉问题词）误识别为商品实体的比例 | ≤8% | 人工抽样判定 |
| 主商品识别准确率（Primary Entity Accuracy） | 在多实体场景中，正确识别主商品/服务对象的比例 | ≥80% | 人工逐条检查 |
| 无实体识别准确率（Negative Detection Accuracy） | 正确识别"确实没有商品实体"的case（不过度抽取） | ≥90% | 人工抽样判定 |

**说明：**
- 以上指标均通过人工抽样评估（如从分类结果中抽200条），不依赖自动计算的 P/R/F1。
- 抽样应覆盖所有L1品类、不同抽取难度、多实体和无实体场景。
- 评估结果用于修订实体词典、优化LLM Prompt和改进排除词词典。

### 11.2 分阶段目标

| 指标 | 阶段一（人工评估） | 阶段二目标 | 阶段三目标 |
|------|-----------------|----------|----------|
| 实体识别召回率 | ≥85%（人工判定） | ≥90% | ≥95% |
| 实体识别精确率 | ≥82%（人工判定） | ≥88% | ≥92% |
| 实体识别 F1 | 不承诺（标注数据不足） | ≥0.85 | ≥0.90 |
| 主实体确定准确率 | ≥80% | ≥88% | ≥90% |
| 无实体识别准确率 | ≥90% | ≥93% | ≥95% |

**F1 > 0.85 的承诺前提：**
- F1 > 0.85 是**阶段二**目标，需要具备以下条件才能评估：
  1. ≥500条实体标注数据（BIO标注或等效格式）
  2. 标注一致性（实体边界一致性）≥0.80
  3. 商品词典覆盖≥500个商品词
  4. 排除词词典经过≥3轮迭代优化
- 阶段一不做 F1 承诺，因为标注数据量不足、实体边界定义尚待验证。

## 12. 品牌识别与归一化

### 12.1 品牌识别定位

**品牌是辅助维度，不是商品品类。** 品牌回答"属于哪个品牌/品牌集团"，商品品类回答"是什么商品/服务"。

**品牌识别的用途：**
- 品牌聚合分析（同一品牌下所有投诉的分布）
- 品牌风险监控（某品牌集中爆发质量/虚假宣传投诉）
- 品牌投诉量统计（按品牌维度查看投诉热点）
- 辅助品类判断（已知品牌 + 已知型号 → 推断品类，但置信度降低）

**品牌识别不能做的事：**
- 用品牌直接替代商品品类（"苹果"不是品类，"手机"才是）
- 当品牌跨多品类且无商品词时，强行判断品类

### 12.2 品牌识别目标

1. **识别品牌提及：** 从标题、内容、商家、商品描述、链接等字段中识别品牌名称
2. **识别多种写法：** 中文名、英文名、大小写变体、别名、简称、系列名、型号名
3. **归一化：** 将不同写法归一到统一品牌标识

**示例：**
- "iPhone"、"苹果"、"Apple"、"APPLE" → 归一化为 `Apple/苹果`
- "HUAWEI"、"华为"、"Mate"、"Pura" → 归一化为 `华为`
- "Nike"、"耐克"、"NIKE" → 归一化为 `Nike/耐克`

### 12.3 品牌实体与其他实体的关系

```
投诉文本: "在抖音买了一个iPhone 15 Pro Max 256G，收到后无法开机"
         │          │         │                │
         │          BRAND     MODEL             ATTRIBUTE
         PLATFORM   (Apple)   (iPhone 15 Pro)   (256G)
```

**实体抽取优先级：**
1. 先抽商品词/服务词（PRODUCT/SERVICE）
2. 再抽品牌词（BRAND）和型号/系列词（MODEL）
3. 识别属性词（ATTRIBUTE）作为补充信息
4. 商品词优先于品牌词：即使品牌置信度高，品类仍以商品词为准

### 12.4 品牌归一化规则

品牌归一化将原始文本中的品牌提及映射到标准品牌名。

| 归一化维度 | 说明 | 示例 |
|-----------|------|------|
| 中英文统一 | 中文名和英文名映射到同一品牌 | Apple ↔ 苹果 |
| 大小写统一 | 大小写变体统一 | apple/APPLE/Apple → Apple |
| 别名映射 | 常见别名/俗称映射 | 果子 → 苹果（需上下文确认） |
| 产品线归属 | 产品线/系列归入母品牌 | iPhone/iPad/Mac → Apple |
| 子品牌关联 | 子品牌记录归属但不合并 | Redmi/红米 → 母品牌=小米，子品牌=Redmi |
| 旧品牌/更名 | 历史品牌名映射 | 荣耀旧机型 → 按当前规则归属 |

**归一化示例（核心品牌）：**

| 原始提及 | 归一化品牌 | 别名类型 |
|---------|----------|---------|
| Apple、苹果、APPLE、apple | Apple/苹果 | 中英文/大小写 |
| iPhone、iPad、MacBook、AirPods、Apple Watch | Apple/苹果 | 产品线 |
| HUAWEI、华为、Huawei | 华为 | 中英文/大小写 |
| Mate、Pura、nova、畅享 | 华为 | 产品线/系列 |
| Xiaomi、小米、MI | 小米/小米集团 | 中英文 |
| Redmi、红米 | 小米/小米集团（子品牌=Redmi） | 子品牌 |
| 米家 | 小米/小米集团（子品牌=米家） | 子品牌 |
| Nike、NIKE、耐克 | Nike/耐克 | 中英文/大小写 |
| Adidas、阿迪达斯、三叶草 | Adidas/阿迪达斯 | 中英文/别名 |

### 12.5 品牌歧义处理

某些词既是品牌又可能是普通商品或日常用语，必须结合上下文判断。

| 歧义词 | 可能含义 | 消歧策略 |
|--------|---------|---------|
| 苹果 | Apple 品牌 vs 水果 | 检查上下文是否有数码/电子相关词；检查是否有"手机""电脑""耳机"等品类词 |
| 小米 | 小米品牌 vs 谷物粮食 | 检查上下文是否有电子产品/家电/智能设备相关词；检查是否在食品/生鲜语境 |
| 荣耀 | 荣耀品牌 vs 普通形容词 | 检查是否作为名词使用（"荣耀手机"→品牌）；检查是否作为形容词（"无上荣耀"→不是品牌） |
| 三星 | Samsung 品牌 vs 无歧义 | 三星几乎无歧义，但需注意"三星堆"等特殊语境 |
| 美的 | 美的品牌 vs 形容词 | 检查是否与家电/电器相关 |
| 联想 | Lenovo 品牌 vs 心理活动 | 检查是否与电脑/电子产品相关 |

**消歧流程：**

```
1. 检测到歧义词
   │
   ▼
2. 搜索上下文（前后10个词范围内）是否有商品/品类词？
   ├─ 有 → 根据品类词判断是品牌还是普通词
   │      "苹果手机" → 品牌=Apple
   │      "苹果坏了" → 品牌=null, 品类=食品生鲜
   └─ 无 → 搜索是否有电子/数码/家电等语境词？
          ├─ 有 → 倾向判断为品牌，降低置信度
          └─ 无 → 标记 need_brand_review=true
```

**品牌歧义对品类判断的影响：**
- 如果只有歧义品牌词，没有商品词 → 必须人工复核，不得高置信度判断品类
- 如果歧义品牌 + 商品词明确 → 以商品词决定品类，品牌作为辅助信息
- 如果跨品类品牌（如"小米"可以是手机/电视/家电）且无型号/商品词 → 降低品类置信度，输出多个候选品类

### 12.6 品牌与品类的关系

**核心原则：商品词优先于品牌词，品牌辅助品类判断。**

| 场景 | 处理方式 | 示例 |
|------|---------|------|
| 商品词明确 + 品牌明确 | 商品词决定品类，品牌为辅助信息 | "iPhone 15手机" → 品类=手机通讯，品牌=Apple |
| 只有品牌+型号，无商品词 | 通过品牌-型号映射推断品类 | "买了一台Mate 60" → 型号映射→手机→品类=手机通讯 |
| 只有品牌，无型号无商品词 | 单一品类品牌→低置信度推断；跨品类品牌→人工复核 | "买了个小米" → 跨品类，need_human_review=true |
| 品牌与商品词冲突 | 以商品词为准 | "苹果水果" → 品类=生鲜水果，品牌=null |
| 品牌是配件/周边 | 品类以实际商品为准 | "苹果手机壳" → 品类=数码配件（不是手机），品牌可能为第三方 |

### 12.7 品牌识别输出JSON

品牌识别结果嵌入实体抽取输出中，作为独立模块：

```json
{
  "detected_brands": [
    {
      "raw_text": "iPhone",
      "normalized_brand": "Apple/苹果",
      "brand_id": "BRAND_APPLE",
      "brand_alias_type": "product_line",
      "confidence": 0.95,
      "source_field": "标题",
      "evidence_text": "iPhone 15无法开机"
    }
  ],
  "main_brand": {
    "normalized_brand": "Apple/苹果",
    "brand_id": "BRAND_APPLE",
    "confidence": 0.95
  },
  "brand_disambiguation": {
    "needed": false,
    "ambiguous_word": null,
    "resolved_by": null,
    "reason": null
  }
}
```

**字段说明：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `detected_brands` | Array | 所有检测到的品牌提及（可能多个） |
| `detected_brands[].raw_text` | String | 原始文本中的品牌提及 |
| `detected_brands[].normalized_brand` | String | 归一化后的品牌名（格式：`English名/中文名`） |
| `detected_brands[].brand_id` | String | 品牌唯一ID（如 `BRAND_APPLE`） |
| `detected_brands[].brand_alias_type` | String | 别名类型：`official_name`/`chinese_name`/`english_name`/`product_line`/`sub_brand`/`alias`/`abbreviation` |
| `detected_brands[].confidence` | Float | 品牌识别置信度（0-1） |
| `detected_brands[].source_field` | String | 品牌提及的来源字段 |
| `detected_brands[].evidence_text` | String | 支持品牌识别的原文片段 |
| `main_brand` | Object | 主品牌（投诉主要涉及的品牌） |
| `main_brand.normalized_brand` | String | 主品牌归一化名称 |
| `main_brand.brand_id` | String | 主品牌ID |
| `main_brand.confidence` | Float | 主品牌置信度 |
| `brand_disambiguation.needed` | Boolean | 是否需要品牌消歧 |
| `brand_disambiguation.ambiguous_word` | String | 歧义词（如"苹果"） |
| `brand_disambiguation.resolved_by` | String | 消歧依据（如"上下文商品词'手机'"） |
| `brand_disambiguation.reason` | String | 消歧说明 |

### 12.8 无品牌时的处理

如果投诉中完全没有品牌信息，品牌相关字段应为空/null，不强制填充。

```json
{
  "detected_brands": [],
  "main_brand": null,
  "brand_disambiguation": {
    "needed": false,
    "ambiguous_word": null,
    "resolved_by": null,
    "reason": "投诉中无品牌提及"
  }
}
```

### 12.9 品牌词典建设（阶段一）

阶段一品牌词典建设方式：

1. **种子品牌词典：** 手动构建 ≥20 个高频消费品牌（参见 `stage1_brand_mapping_seed.json`）
2. **从投诉数据挖掘：** 使用 LLM 从 24,713 条投诉标题中抽取高频品牌词
3. **品牌别名扩展：** 为每个品牌补充中文名、英文名、常见产品线/系列名
4. **品牌-品类映射：** 为每个品牌标注常见商品品类，用于辅助品类判断
5. **歧义标注：** 标注品牌是否具有跨品类歧义、是否为日常用语歧义

详细设计见 `brand_mapping_design.md`。

### 12.10 品牌识别评估指标（阶段一）

阶段一品牌识别采用人工抽样评估：

| 指标 | 定义 | 阶段一目标 |
|------|------|----------|
| 品牌识别覆盖率 | 有品牌提及的投诉中，至少识别出一个品牌的比例 | ≥80% |
| 品牌归一化准确率 | 品牌归一化正确的比例（同一品牌的不同写法归一到正确品牌） | ≥85% |
| 品牌歧义消解准确率 | 歧义品牌词（苹果/小米/荣耀等）正确判断的比例 | ≥75% |
| 品牌误识率 | 将非品牌词误识别为品牌的比例 | ≤5% |
| 品牌-品类冲突率 | 品牌识别结果与商品品类不一致的样本比例 | ≤3% |
