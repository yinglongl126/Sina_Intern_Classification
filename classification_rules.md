# 商品/服务品类分类规则（第二版）

## 1. 核心分类原则

### 1.1 品类回答"对象是什么"，问题回答"原因是什么"

**红线规则：投诉问题词不能作为商品品类。**

| 投诉文本 | ✅ 商品/服务品类（本项目） | ❌ 不是品类 |
|---------|------------------------|-----------|
| "显卡温度过高黑屏" | 数码电子 > 电脑整机与硬件 | 质量问题 |
| "羽绒服掉毛严重" | 服饰鞋包 > 女装/男装 | 商品质量问题 |
| "手机屏幕有坏点" | 数码电子 > 手机通讯 | 质量问题 |
| "蛋糕变质" | 食品生鲜 > 烘焙糕点 | 食品安全问题 |
| "会员无法使用" | 虚拟商品 > 会员与订阅服务 | 服务故障 |
| "退款一直不到账" | **无法确定品类，标记人工复核** | 退款纠纷 |

### 1.2 实体抽取前置

**必须先做实体抽取，再分类。** 如果实体抽取模块没有抽到任何商品/服务实体，不能强行分类，必须进入人工复核。

流程：投诉文本 → 实体抽取 → 按实体做品类匹配 → 分类输出

详细规则见 `entity_extraction_design.md`。

### 1.3 主品类原则（多商品时）

1. **标题首次提及优先** — 标题中第一个出现的商品/服务词
2. **金额关联优先** — 金额描述最接近的商品
3. **问题关联优先** — 在"问题"描述附近出现的商品
4. **位置权重优先** — 越靠前出现的权重越高

**示例：**
> "买了一台电脑和鼠标，电脑有问题"
> → 主品类：数码电子 > 电脑整机与硬件
> → 辅品类：[数码电子 > 电脑外设与网络设备（鼠标）]

## 2. 服务对象类型判断规则

```
投诉对象是实物商品 → PHYSICAL
投诉对象是无实体的数字商品 → VIRTUAL
投诉对象是需要线下消费的服务 → LOCAL
投诉对象是平台账号/店铺/直播/物流功能 → PLATFORM
投诉对象是金融产品/保险产品 → OTHER
```

**实用判断法：** 
- 需要用快递寄送的东西 → PHYSICAL
- 可以下载/在线使用/通过账号获取 → VIRTUAL
- 需要去实体店/线下场所 → LOCAL
- 是电商平台本身的功能/服务 → PLATFORM

## 3. 关键边界规则

### 3.1 物流/快递 — 什么时候是品类，什么时候不是

| 投诉场景 | 是否作为品类 | 处理 |
|---------|------------|------|
| "快递丢失了"（投诉对象是快递服务本身） | ✅ 是 | 平台服务 > 物流配送服务 |
| "买的手机，快递太慢"（投诉对象是手机，快递是附属问题） | ❌ 不是 | 品类=手机，快递慢是辅助标签 |
| "物流信息一直不更新" | ✅ 是 | 平台服务 > 物流配送服务 |
| "衣服寄错地址了"（投诉对象是衣服，物流出错） | ❌ 不是 | 品类=衣服 |

**判断标准：** 投诉的核心焦点是"物流服务本身"还是"某个商品的物流环节"。

### 3.2 外卖/配送 — 什么时候是品类

| 投诉场景 | 是否作为品类 | 分类 |
|---------|------------|------|
| "外卖送了一个小时还没到" | ✅ 是 | 本地生活服务 > 餐饮外卖 |
| "外卖里的菜是坏的" | ✅ 是 | 本地生活服务 > 餐饮外卖（菜品质量） |
| "在拼多多买了菜，配送太慢"（多多买菜） | 看情况 | 如果投诉焦点是菜品→食品生鲜；如果焦点是配送→平台服务>物流配送服务 |

### 3.3 运费险 — 品类还是金融产品

| 投诉场景 | 是否作为品类 | 分类 |
|---------|------------|------|
| "运费险不赔付" | ✅ 是 | 金融支付与保险服务 > 保险产品 |
| "买了运费险但退不了" | ✅ 是 | 金融支付与保险服务 > 保险产品 |
| "退货了，运费险没用上"（核心诉求是退货） | ❌ 不是 | 品类=对应商品，运费险是辅助标签 |

**运费险本身是一个保险产品**，如果投诉的核心是运费险的购买/理赔问题，它就是被投诉的服务对象。

### 3.4 优惠券/红包/卡券

| 投诉场景 | 是否作为品类 | 分类 |
|---------|------------|------|
| "优惠券无法使用" | ✅ 是 | 虚拟商品 > 数字卡券与充值 |
| "平台红包过期了不给退" | ✅ 是 | 虚拟商品 > 数字卡券与充值 |
| "买手机送的优惠券用不了" | ❌ 不是 | 品类=手机，优惠券是附属 |

### 3.5 直播间

| 投诉场景 | 是否作为品类 | 分类 |
|---------|------------|------|
| "直播间被限流/封禁"（投诉对象是直播间功能） | ✅ 是 | 平台服务 > 直播与内容服务 |
| "在直播间买的东西是假的"（投诉对象是商品） | ❌ 不是 | 品类=对应商品，直播间是购买渠道 |
| "主播虚假宣传"（投诉对象是主播行为） | 看情况 | 如果涉及具体商品，品类=该商品；如果纯主播行为，归入平台服务 |

### 3.6 账号/店铺

| 投诉场景 | 是否作为品类 | 分类 |
|---------|------------|------|
| "账号无故被封禁" | ✅ 是 | 平台服务 > 账号服务 |
| "店铺保证金被扣" | ✅ 是 | 平台服务 > 店铺与商家服务 |
| "账号被盗，里面的优惠券被用了" | ✅ 是 | 平台服务 > 账号服务（主），辅=虚拟商品>数字卡券 |

### 3.7 支付问题

| 投诉场景 | 是否作为品类 | 分类 |
|---------|------------|------|
| "支付宝/微信支付无法使用" | ✅ 是 | 金融支付 > 支付与账户 |
| "买东西付款了但商家说没收到" | ❌ 不是 | 品类=对应商品，支付异常是辅助标签 |
| "花呗/白条扣款异常" | ✅ 是 | 金融支付 > 消费金融 |

### 3.8 保证金/入驻费

| 投诉场景 | 是否作为品类 | 分类 |
|---------|------------|------|
| "平台不退保证金" | ✅ 是 | 平台服务 > 店铺与商家服务 |
| "入驻费被骗了" | ✅ 是 | 平台服务 > 店铺与商家服务 |

## 4. 特殊场景分类规则

### 4.1 只有品牌没有商品名

**场景：** "苹果有问题"、"华为质量差"

**规则：**
1. 如果品牌可明确指向最常见品类（"苹果"→智能手机，"格力"→空调），按最常见品类分类，置信度降低至0.70-0.80
2. 如果品牌跨多品类（"小米"、"三星"、"索尼"），**标记 need_human_review=true**，所有可能品类作为辅分类
3. 在 reason 中说明推断假设

### 4.2 只有店铺没有商品名

**规则：**
1. 店铺名含品类线索（"XX手机旗舰店"）→ 使用店铺推断
2. 店铺名无线索（"XX严选"）→ 从正文搜索商品词
3. 完全没有 → need_human_review=true

### 4.3 商品名称模糊

**规则：**
1. 从上下文搜索材质、用途、场景等线索
2. 从商家名称推断
3. 无法推断 → need_human_review=true，L1标记为可能候选+低置信度

### 4.4 虚拟商品和实物商品混合

**场景：** "买了手机，送的会员不能用"

**规则：**
1. 主投诉对象 = 购买的主要商品（手机）
2. 次要对象 = 赠品/附属服务（会员）
3. 如果两者同等重要 → 标记为跨品类，need_human_review=true

### 4.5 无商品实体：投诉只有"退款"、"质量差"等

**规则：** 必须进入人工复核。
- 归入 "其他/待人工复核 > 暂无法归类"
- 不能强行分到任何实物商品品类
- 在 human_review_reason 中标记为 "NO_PRODUCT_ENTITY"

## 5. 人工复核触发规则（10条）

| 编号 | 触发条件 | 说明 |
|------|---------|------|
| HR-1 | 置信度 < 0.60 | 模型对分类不确定 |
| HR-2 | 置信度 0.60-0.75 | 偏低，建议人工确认 |
| HR-3 | 多L1候选≥3个且得分接近（差距<0.10） | 无法确定主品类 |
| HR-4 | 实体抽取失败（无商品/服务实体） | 标题和正文都找不到实体 |
| HR-5 | 品牌跨≥3个品类且上下文无法消歧 | 如"小米" |
| HR-6 | 投诉涉及疑似新品类 | 现有L2无法覆盖 |
| HR-7 | 涉及违禁品/限制品 | 医药、金融合规 |
| HR-8 | 投诉内容极短（<10字）且信息不足 | 标题如"投诉"、"退款" |
| HR-9 | 多商品/服务同等重要，无法区分主次 | 跨品类投诉 |
| HR-10 | 模型分类结果与品类体系约束冲突 | 如PHYSICAL被分到VIRTUAL品类 |

---

## 6. 降低"其他/待人工复核"比例的策略

### 6.1 阶段一（当前目标：<30%进入人工复核）

- 大模型覆盖主流品类
- 商品词典覆盖高频商品词
- 仅真正无法判断的才进入人工复核

### 6.2 阶段二（目标：<20%）

- 商品词典扩大覆盖
- 轻量分类模型接管高频品类
- 长尾仍由LLM处理

### 6.3 阶段三（目标：<15%）

- 模型覆盖成熟
- 高频品类自动准确率>90%
- 剩余的"其他"主要是真正的新品类和疑难case

### 6.4 四个降低策略

1. **扩大商品词典** — 每发现一个新的无法匹配的商品词，就加入词典
2. **品牌-品类映射库** — 减少"品牌歧义→人工复核"的比例
3. **人工反馈闭环** — 复核后的正确分类直接入库
4. **主动学习** — 优先标注模型最不确定的case，让每一条标注都高效

## 7. 分类结果校验

### 7.1 逻辑一致性
- L2必须属于L1，品类编码必须匹配
- product_service_type必须与L1类型一致

### 7.2 置信度合理性
- 仅有标题无商品名 → 置信度不应>0.5
- 标题+正文都明确提到同一商品 → 可>0.85
- 品牌歧义且无上下文 → 置信度应<0.70

### 7.3 输出证据要求
- evidence_spans 必须包含至少一个商品/服务实体
- 如果没有实体，need_human_review 必须为 true
- reason 必须用自然语言解释分类依据

## 8. 品牌相关分类规则

### 8.1 品牌识别不是简单关键词分类

**核心规则：关键词命中品牌词典只是候选生成（candidate），不是最终识别结果（final）。**

品牌识别的四步流程：
1. **候选生成：** 品牌词典/别名/产品线/型号命中 → 生成候选品牌列表
2. **候选过滤：** 上下文消歧 → 排除假阳性（如"苹果水果""美的风景"）
3. **置信度计算：** 按命中类型赋基础置信度 + 上下文调节
4. **复核决策：** 低置信度/歧义/冲突 → need_brand_review=true

**关键约束：**
- `苹果`、`小米`、`荣耀`、`美的`、`联想` 等歧义词必须做上下文消歧
- `MI`、`Mac`、`Mate` 等短别名（≤3字符）不能单独高置信度触发品牌
- 产品线命中（iPhone、iPad、AirPods、MacBook）比短别名更可靠
- 型号命中（RTX 5090、Mate 60）需通过型号-品牌-品类映射确认
- 平台名（淘宝、京东、拼多多）不是商品品牌
- 商家名中的"官方客服""售后客服"不是品牌

### 8.2 商品词优先于品牌词

**核心规则：品类由商品/服务实体决定，不是由品牌决定。**

| 优先级 | 规则 | 说明 |
|--------|------|------|
| 1 | 商品词明确 → 以商品词定品类 | 无论品牌是什么 |
| 2 | 品牌词 + 型号/产品线 → 可推断品类 | 仅当没有明确的商品词时 |
| 3 | 只有品牌词 → 按品牌-品类映射推断 | 置信度降低，跨品类品牌必须人工复核 |
| 4 | 品牌词和商品词冲突 → 以商品词为准 | 品牌不得覆盖商品判断 |

### 8.3 品牌置信度规则

**重要：`brand_confidence` 是最终品牌置信度（final_brand_confidence），不是词典命中置信度（candidate_match_confidence）。**

| 命中类型 | 示例 | 候选置信度 | 最终置信度 | 处理 |
|---------|------|-----------|-----------|------|
| 品牌名 + 品牌上下文 | "华为手机"、"Nike鞋" | 0.90-0.98 | 0.90-0.98 | 可自动识别 |
| 产品线精确命中 | iPhone、iPad、AirPods | 0.88 | 0.85-0.95 | 可推断品牌和品类 |
| 型号/系列精确命中 | RTX 5090、Mate 60 | 0.88 | 0.80-0.92 | 需型号→品牌→品类映射 |
| 别名 + 品牌上下文 | "耐克球鞋" | 0.88-0.93 | 0.80-0.95 | 上下文校验通过 |
| 歧义词 + 品牌上下文 | "买了一台苹果手机" | 0.55 | 0.80-0.95 | 消歧通过后可提升 |
| 歧义词 + 无明确上下文 | "苹果有问题" | 0.55 | 0.40-0.55 | 无法消歧，必须复核 |
| 歧义词 + 非品牌上下文 | **"昭通苹果尺寸不符"** | 0.55 | **0** | **强制判定为非品牌** |
| 短别名 + 品牌上下文 | "Mac电脑" | 0.65 | 0.70-0.85 | 有上下文可适度提升 |
| 短别名 + 无上下文 | "买了个Mac" | 0.65 | 0.50-0.60 | 必须复核 |
| 模糊拼写/疑似型号 | "mete80promax" | 0.55 | 0.60-0.75 | 疑似型号错写，必须复核 |
| 店铺名命中 | XX旗舰店 | 0.60-0.85 | 0.60-0.85 | 辅助证据 |

**置信度调节规则：**
- 品牌与商品品类一致 → 上调 +0.05~0.10
- 歧义词 + 明确非品牌上下文 → **强制置零，confidence=0，不识别为品牌**
- 多证据交叉验证（标题+内容+商家）→ 上调 +0.05~0.10
- 只有品牌没有商品词 → 上限 ≤0.75
- 跨品类品牌 + 只有品牌词 → need_brand_review=true，置信度 ≤0.60
- 短别名 + 无上下文 → need_brand_review=true，置信度 ≤0.60
- 歧义词 + 无法消歧 → need_brand_review=true，置信度 ≤0.55
- 仅在商家名出现 → 下调 -0.10~-0.20
- need_brand_review=TRUE 时，除非有明确产品线/型号证据，置信度不得高于 0.75

**歧义词非品牌判定示例：**

| 投诉文本 | 判定 | 原因 |
|---------|------|------|
| "昭通苹果尺寸不符，70-75mm" | NOT_BRAND | 昭通+尺寸+mm → 水果语境 |
| "云南苹果新鲜现摘" | NOT_BRAND | 云南+新鲜+现摘 → 水果语境 |
| "买了一台苹果手机" | BRAND=Apple | 手机 → 品牌语境 |
| "买了两斤小米煮粥" | NOT_BRAND | 两斤+煮粥 → 粮食语境 |
| "小米电视黑屏" | BRAND=小米 | 电视 → 品牌语境 |
| "风景很美的" | NOT_BRAND | 风景+的 → 形容词语境 |
| "美的空调不制冷" | BRAND=美的 | 空调 → 品牌语境 |

### 8.4 品牌词辅助品类判断

品牌和型号/产品线可以在没有明确商品词时辅助品类推断。

**示例：**

| 投诉文本 | 品牌识别 | 品类推断 | 置信度 |
|---------|---------|---------|--------|
| "买了一台Mate 60" | brand=华为, product_line=Mate, model=Mate 60 | 通过 Mate 产品线→手机→数码电子>手机通讯 | 0.80 |
| "iPhone 15无法开机" | brand=Apple, product_line=iPhone, model=iPhone 15 | 通过 iPhone 产品线→手机→数码电子>手机通讯 | 0.85 |
| "新买的AirPods有杂音" | brand=Apple, product_line=AirPods | 通过 AirPods→无线耳机→数码电子>影音娱乐 | 0.85 |
| "买了个5090" | brand=NVIDIA, model=RTX 5090 | 通过型号5090→显卡→数码电子>电脑整机与硬件 | 0.85 |

### 8.5 只有品牌无商品词时的处理

| 品牌类型 | 处理方式 | 示例 |
|---------|---------|------|
| 单一品类品牌 | 可低/中置信度推断品类（0.60-0.80） | "买了个NVIDIA" → 品类=显卡 |
| 跨品类品牌 | **必须人工复核**，可输出候选品类列表 | "买了个小米" → need_human_review=true |
| 歧义品牌 | **必须人工复核**，先判断是不是品牌 | "买了个苹果" → 先判断是水果还是Apple |
| 未知品牌 | 标记为新品牌候选，人工复核 | "买了个XX新品牌" → new_category_candidate=true |

### 8.6 品牌与品类冲突时的处理

**规则：当品牌识别的默认品类与实际商品词冲突时，以商品词为准。**

| 投诉文本 | 品牌默认品类 | 实际商品词 | 最终品类 | 品牌处理 |
|---------|-----------|----------|---------|---------|
| "苹果水果坏了" | 数码电子（Apple默认） | 水果（实际上不是品牌） | 食品生鲜>生鲜水果 | brand=null |
| "小米手机壳" | 手机通讯（小米默认） | 手机壳（配件） | 数码电子>手机通讯（手机配件） | brand=小米，但品类≠手机 |
| "华为电视黑屏" | 手机通讯（华为默认） | 电视 | 家用电器>大家电 | brand=华为，品类按商品词 |
| "苹果耳机不是正品" | 手机通讯（Apple默认） | 耳机 | 数码电子>影音娱乐与游戏设备 | brand=Apple，品类按商品词 |

**关键判断逻辑：**
1. 先抽取商品词，确定品类
2. 再识别品牌
3. 如果品牌识别的默认品类与商品词品类不一致 → 以商品词品类为准
4. 如果出现"品牌+配件/周边"模式，品类=配件而非品牌主品类

### 8.7 品牌-品类交叉验证

分类完成后，进行品牌-品类合理性校验：

```
IF brand_info.main_brand IS NOT NULL AND product_category IS NOT 98:
    FOR EACH common_category IN brand.common_categories:
        IF category_matches(common_category, product_category):
            → 品牌-品类一致，通过
    IF NO match:
        → 检查是否合规：
          - 品牌+配件/周边 → 品类可能是配件（合理）
          - 品牌跨品类应用 → 可能是低频品类（合理但降低置信度）
          - 明显冲突 → 标记 need_human_review
```

**冲突示例：**
- 品牌=NVIDIA，品类=食品生鲜 → 明显冲突，进入人工复核
- 品牌=Apple，品类=手机通讯 → 一致，通过
- 品牌=Apple，品类=数码电子>电脑外设 → 可能是苹果配件，合理但不常见，置信度稍降

### 8.8 品牌聚合不影响商品品类统计

**重要：品牌维度的聚合分析不影响商品品类的独立统计。**

- 按品牌统计投诉量：统计的是"该品牌下所有投诉"，不改变每条投诉的商品品类
- 品牌投诉分布不等于品类投诉分布
- 同一品牌的投诉可能分布在多个品类
- 品牌-品类交叉统计可以提供额外洞察（如"小米品牌在手机品类的投诉量 vs 在家电品类的投诉量"）

### 8.9 品牌人工复核触发规则（6条）

| 编号 | 触发条件 | 说明 |
|------|---------|------|
| BR-1 | 品牌歧义词且上下文无法消歧 | 如"苹果"在无商品词上下文中无法判断是水果还是Apple |
| BR-2 | 多品牌提及且无法确定主品牌 | 如投诉中提到多个品牌（"买了华为手机和苹果耳机"） |
| BR-3 | 品牌与品类冲突且无法合理解释 | 如品牌=NVIDIA但品类=食品，或品牌=Apple但品类=服饰 |
| BR-4 | 疑似新品牌 | 品牌不在已知品牌词典中，且频次低 |
| BR-5 | 品牌归一化低置信度 | 品牌匹配不确定（如模糊匹配得分低） |
| BR-6 | 只有跨品类品牌无商品词 | 如"买了个小米"、"买了个三星" |
