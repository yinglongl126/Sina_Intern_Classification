# 阶段一小样本试标任务说明

## 0. 文档说明

本文档定义第一阶段的小样本试标任务。目标是：在正式大规模标注之前，用 100 条投诉样本验证品类体系可用性、标注流程可行性和标注指南清晰度。

**试标不是一次性任务。** 如果发现品类体系缺口、标注指南歧义或标注一致性不达标，需要修订体系/指南后追加试标。

---

## 1. 试标目标

1. **验证品类体系：** 100 条样本是否都能在 `stage1_taxonomy.json` 中找到合适的 L1/L2 归属（或合理归入兜底类）。
2. **验证实体抽取流程：** 从每条投诉中抽取商品/服务实体，确认实体抽取规则是否足够清晰。
3. **验证标注指南：** 标注员能否按 `stage1_annotation_guideline.md` 独立完成标注，且两人标注一致性达标。
4. **暴露问题：** 发现品类缺口、边界模糊案例、实体抽取困难 case、标注指南歧义点。
5. **积累参考样本：** 为后续人工标注和模型训练提供第一批参考样本。

---

## 2. 输入数据

### 2.1 数据来源

原始 Excel 文件：`data/电商平台25年12月投诉数据.xlsx`（24,713 条）

### 2.2 试标样本

从原始 Excel 抽取 100 条，生成 `stage1_pilot_samples.csv`。

### 2.3 输入字段

| 字段 | 来源列 | 说明 |
|------|--------|------|
| sample_id | 自动生成 | 试标样本编号，S001-S100 |
| complaint_id | 单号 | 原始投诉单号 |
| platform | 服务名称 | 投诉提交的电商平台（淘宝商城/拼多多/京东/抖音/得物等） |
| title | 标题 | 投诉标题，商品/服务信息的主要来源 |
| content | 内容 | 投诉正文，补充商品细节、品牌、型号 |
| merchant | 商家 | 商家名称，可能含品类线索（如"XX数码专营店"） |
| demand | 诉求 | 消费者诉求（退货退款/赔偿/改善服务等） |
| issue | 问题 | 投诉问题类型（原始字段，如"未履行7天无理由退款"） |
| amount | 金额 | 涉及金额（元） |
| link | 链接 | 投诉详情链接（辅助参考，非主要抽取来源） |

## 3. 试标流程

### 3.1 整体流程

```
Step 0: 阅读 stage1_annotation_guideline.md + stage1_taxonomy.json
Step 1: 逐条阅读投诉样本
Step 2: 抽取商品/服务实体（记录实体词、实体类型、来源字段）
Step 3: 判断 service_type（PHYSICAL/VIRTUAL/LOCAL/PLATFORM/OTHER）
Step 4: 在 stage1_taxonomy.json 中选择 L1 → L2
Step 5: 评估置信度
Step 6: 判断是否需要人工复核
Step 7: 记录分类依据（evidence_text）
Step 8: 记录备注（notes）
```

### 3.2 标注流程（详细）

```
1. 先读标题
   └─ 有商品/服务词？ → 记录实体 → Step 2
   └─ 无商品/服务词？ → 读内容首段
        ├─ 有实体 → 记录+标记来源 → Step 2
        └─ 无实体 → 读全文+商家名
             ├─ 有实体（含商家线索） → 记录+标注"通过商家推断" → Step 2
             └─ 无实体 → 标记 extraction_status=FAILED → Step 7

2. 判断 service_type
   └─ 需要快递寄送的东西 → PHYSICAL
   └─ 可下载/在线使用/通过账号获取 → VIRTUAL
   └─ 需要去线下实体场所 → LOCAL
   └─ 电商平台本身的功能/服务 → PLATFORM
   └─ 以上都不属于 → OTHER

3. 在 stage1_taxonomy.json 中查品类
   └─ 按抽取的实体词匹配最合适的 L2
   └─ 确定该 L2 所属的 L1
   └─ 如果无法匹配任何 L2 → 归入对应的 XX-99（或 98-01/98-02）

4. 评估置信度
   └─ ≥0.85: 标题+内容都明确提到同一商品/服务
   └─ 0.75-0.84: 标题或内容有明确商品词
   └─ 0.60-0.74: 仅通过间接线索推断
   └─ <0.60: 信息不足，高度不确定
   └─ 0.0: 无任何商品/服务实体

5. 判断是否需人工复核
   └─ 参考 classification_rules.md 中的 HR-1~HR-10 规则
```

## 4. 输出格式

### 4.1 试标输出文件

建议使用 CSV（UTF-8 编码），方便 Excel 打开和后续处理。

或者使用 JSONL，每行一个 JSON 对象，方便程序读取。

### 4.2 输出字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| sample_id | string | 是 | 试标样本编号，与输入一致（S001-S100） |
| complaint_id | string | 是 | 原始投诉单号 |
| title | string | 是 | 原始标题（便于对照检查） |
| content_summary | string | 否 | 内容摘要（可选，方便快速浏览，不超过100字） |
| extraction_status | string | 是 | SUCCESS（抽到≥1个商品/服务实体）或 FAILED（未抽到） |
| detected_entities | string | 是 | 抽取到的所有商品/服务实体，用`；`分隔。格式：`实体词[TYPE][来源字段]`，如 `显卡[PRODUCT][标题]；5090[MODEL][内容]` |
| main_entity | string | 是（抽到时） | 主商品/服务实体词，如 `显卡`。无实体时填 `N/A` |
| main_entity_type | string | 是（抽到时） | 主实体类型：PRODUCT / BRAND / SERVICE / MODEL。无实体时填 `N/A` |
| product_service_type | string | 是 | PHYSICAL / VIRTUAL / LOCAL / PLATFORM / OTHER |
| l1_code | string | 是 | 一级品类编码，如 `06`。必须来自 stage1_taxonomy.json |
| l1_name | string | 是 | 一级品类名称，如 `数码电子` |
| l2_code | string | 是 | 二级品类编码，如 `06-02`。必须来自 stage1_taxonomy.json |
| l2_name | string | 是 | 二级品类名称，如 `电脑整机与硬件` |
| l3_code | string | 否 | 三级品类编码（阶段一填 `null` 或留空） |
| l3_name | string | 否 | 三级品类名称（阶段一填 `null` 或留空） |
| confidence | float | 是 | 分类置信度，0.00-1.00 |
| need_human_review | bool | 是 | TRUE / FALSE |
| human_review_reason | string | 是（需要时） | 触发人工复核的原因代码，如 `HR-4: 无商品实体`。不需要时填 `N/A` |
| evidence_text | string | 是 | 支持分类的关键原文片段，用引号标出，如 `"显卡温度过高"—标题` |
| annotator_id | string | 是 | 标注员编号 |
| annotation_date | string | 是 | 标注日期，格式 YYYY-MM-DD |
| detected_brands | string | 否 | 抽取到的品牌实体，用`；`分隔。格式：`原始文本→归一化品牌[BRAND_ID]`，如 `iPhone→Apple/苹果[BRAND_APPLE]；5090→NVIDIA/英伟达[BRAND_NVIDIA]`。无品牌时填 `N/A` |
| main_brand | string | 否 | 主品牌归一化名称，如 `Apple/苹果`。无品牌时填 `N/A` |
| normalized_brand_name | string | 否 | 主品牌归一化名称（与main_brand一致），如 `Apple/苹果` |
| brand_id | string | 否 | 主品牌ID，如 `BRAND_APPLE`。无品牌时填 `N/A` |
| brand_confidence | float | 否 | 品牌识别置信度，0.00-1.00。无品牌时填 `0.00` |
| product_line | string | 否 | 产品线/系列，如 `iPhone`、`Mate`、`GeForce RTX`。无法确定时填 `N/A` |
| brand_review_required | bool | 是 | TRUE / FALSE，品牌识别结果是否需要人工复核 |
| brand_review_reason | string | 否（需要时） | 品牌人工复核原因代码，如 `BR-1: 苹果歧义无法消解`。不需要时填 `N/A` |
| notes | string | 否 | 标注过程中遇到的问题、不确定的地方、对品类体系或品牌词典的建议 |

### 4.3 输出示例（CSV 行）

```csv
S001,17390890267,买了两万多的显卡，温度过高一直黑屏不能使用,在淘宝购买5090显卡，安装后温度过高黑屏,SUCCESS,显卡[PRODUCT][标题];5090[MODEL][内容],显卡,PRODUCT,PHYSICAL,06,数码电子,06-02,电脑整机与硬件,,,0.95,FALSE,N/A,""显卡""—标题；""5090""—内容",A01,2025-06-12,5090→NVIDIA/英伟达[BRAND_NVIDIA],NVIDIA/英伟达,NVIDIA/英伟达,BRAND_NVIDIA,0.90,GeForce RTX,FALSE,N/A,
```

### 4.4 无实体时的输出示例

```csv
S099,17390890XXX,要求退款,等了一个月还没退款,FAILED,N/A,N/A,N/A,OTHER,98,其他/待人工复核,98-01,暂无法归类,,,0.00,TRUE,HR-4: 无商品实体,无,A01,2025-06-12,N/A,N/A,N/A,N/A,0.00,N/A,FALSE,N/A,标题和内容仅含"退款"，无商品描述
```

## 5. 抽样策略

### 5.1 样本组成（100条）

| 类型 | 数量 | 说明 |
|------|------|------|
| 随机样本 | 40 条 | 从全部 24,713 条中随机抽取，保证基本代表性 |
| 高金额样本 | 15 条 | 金额 ≥ 5000 元，高金额投诉通常信息更完整 |
| 标题含明确商品词 | 20 条 | 标题中直接包含显卡/手机/衣服/食品等明确商品词，检验"简单case"的分类效果 |
| 标题不含明确商品词 | 15 条 | 标题如"要求退款""太差了""投诉"等，检验困难case和兜底分类 |
| 平台覆盖样本 | 10 条 | 确保覆盖不同平台：淘宝/天猫、拼多多、京东、抖音、得物、其他 |

### 5.2 平台覆盖目标

| 平台（服务名称） | 最少样本数 |
|----------------|----------|
| 淘宝商城 | 15 |
| 拼多多 | 20 |
| 京东 | 10 |
| 抖音 | 8 |
| 得物 | 3 |
| 其他（天猫/快手/小红书等） | 至少覆盖3个不同平台 |

### 5.3 不要求但建议关注

- 尽量覆盖不同商品形态：实物商品、虚拟商品、服务类投诉
- 尽量包含至少 2-3 条"多商品/服务混合"的投诉
- 尽量包含至少 2-3 条"品牌歧义"的投诉（如"苹果""小米""三星"）

---

## 6. 试标验收标准

### 6.1 必须满足

- [ ] 100 条样本全部完成标注（无跳过、无空白）
- [ ] 每条样本的 L1/L1_code 必须来自 `stage1_taxonomy.json`（不允许凭空编造品类）
- [ ] 每条样本都有实体抽取结果（`detected_entities` + `extraction_status`），或明确标记为 `FAILED`
- [ ] 无商品实体的样本不得强行分类到具体商品品类，必须归入 `98-01`
- [ ] 低置信度（<0.60）样本必须标记 `need_human_review=TRUE`
- [ ] 有品牌提及的样本必须填写品牌相关字段（`detected_brands`、`main_brand`、`brand_id`、`brand_confidence`）
- [ ] 品牌歧义样本（如"苹果"在无商品词上下文中）必须标记 `brand_review_required=TRUE`
- [ ] 品牌-品类冲突样本（如品牌=Apple但品类=食品）必须在 `brand_review_reason` 中说明
- [ ] 至少 20 条样本有双人独立标注结果（用于计算标注一致性）
- [ ] 输出文件编码为 UTF-8

### 6.2 建议满足

- [ ] L1 分布覆盖至少 12 个不同 L1 品类
- [ ] 标注员在 `notes` 字段记录的问题不少于 10 条
- [ ] 如有新品类候选，在 `notes` 或独立清单中记录

## 7. 试标后统计指标

### 7.1 分类分布统计

| 指标 | 计算方式 |
|------|---------|
| L1 分布 | 每个 L1 品类下的投诉数量及占比 |
| L2 分布 | 每个 L2 品类下的投诉数量及占比（含兜底 L2） |
| 无法抽取商品实体比例 | `extraction_status=FAILED` 的样本占比 |
| 进入人工复核比例 | `need_human_review=TRUE` 的样本占比 |
| 各置信度区间分布 | ≥0.85 / 0.75-0.84 / 0.60-0.74 / <0.60 各区间占比 |

### 7.2 实体抽取统计

| 指标 | 计算方式 |
|------|---------|
| 高频商品实体 TOP20 | 按 `main_entity` 频次排序 |
| 实体类型分布 | PRODUCT / BRAND / SERVICE / MODEL 各类型占比 |
| 多实体投诉比例 | `detected_entities` 包含 2 个及以上实体的样本占比 |
| 商家线索使用比例 | 通过商家名称推断品类的样本占比 |

### 7.3 问题与缺口统计

| 指标 | 计算方式 |
|------|---------|
| 高频疑难样本类型 | 按 `human_review_reason` 分类统计 |
| Taxonomy 缺口清单 | 归入 XX-99 或 98-02 的样本，分析是否有可新增的 L2 |
| 边界模糊案例清单 | 标注员在 `notes` 中记录的归属争议 case |
| 标注指南歧义点 | 双人标注不一致的样本，分析分歧原因 |

### 7.4 品牌识别统计

| 指标 | 计算方式 |
|------|---------|
| 品牌识别覆盖率 | `detected_brands` 非空的样本占比 |
| 高频品牌 TOP20 | 按 `main_brand` 频次排序 |
| 品牌歧义样本数量 | `brand_review_required=TRUE` 且原因为 BR-1 的样本数 |
| 只有品牌无商品词的样本数量 | 品牌已识别但 `extraction_status=FAILED` 的样本数 |
| 品牌-品类冲突样本数量 | 品牌默认品类与实际商品品类不一致的样本数 |
| 新品牌候选清单 | 在品牌词典中未找到的品牌提及列表 |
| 产品线识别率 | `product_line` 非空的样本占比 |
| 多品牌投诉比例 | `detected_brands` 包含 2 个及以上品牌的样本占比 |

### 7.5 标注一致性

| 指标 | 目标 | 计算方式 |
|------|------|---------|
| L1 标注一致率 | ≥85% | 双人标注 L1 一致的比例 |
| L2 标注一致率 | ≥75% | 双人标注 L2 一致的比例 |
| Kappa 系数 | ≥0.70 | 对双人标注子集计算 Cohen's Kappa |
| 实体抽取一致率 | ≥80% | 双人标注的主实体一致比例 |
| 品牌标注一致率 | ≥80% | 双人标注的主品牌一致比例 |

## 8. 试标工具建议

### 8.1 简易方案（Excel）

- 使用 Excel 打开 `stage1_pilot_samples.csv`
- 新增标注列（见第 4 节输出字段）
- 品类编码可借助下拉菜单（数据验证→列表→引用 taxonomy）
- 适合 1-2 人小规模试标

### 8.2 进阶方案（标注工具）

- 使用 Label Studio / Doccano 等开源标注工具
- 预先导入品类体系作为标签选项
- 支持双人标注和一致性自动计算
- 适合持续标注和迭代

---

## 9. 试标后行动

1. **统计指标：** 按第 7 节要求统计全部指标。
2. **问题清单：** 汇总标注员在 `notes` 中记录的所有问题。
3. **品类体系修订：** 根据 taxonomy 缺口和边界模糊案例，修订 `stage1_taxonomy.json` 和 `category_taxonomy_design.md`。
4. **标注指南修订：** 根据标注员反馈和双人标注不一致点，修订 `stage1_annotation_guideline.md`。
5. **Prompt 优化：** 根据高频错误模式，优化 LLM 实体抽取和分类 Prompt。
6. **实体词典补充：** 从试标中发现的未覆盖商品词加入商品词典。
7. **决定是否扩量：** 如果试标效果达标（L1 一致率 ≥85%，Kappa ≥0.70），启动正式标注（Phase 1 第 2 周的 1500 条）。如果不达标，先修问题再追标 50 条。
