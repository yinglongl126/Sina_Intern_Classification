# 商品/服务品类体系设计（第二版 — 分阶段可落地）

## 0. 本文档说明

本文档是 `category_taxonomy_design.md` 的第二版。与第一版的主要变化：

1. **从"一上来就做600个L3"改为"分三阶段逐步细化"**
2. **删除了混入品类树的投诉问题类标签：退款纠纷、客服响应、售后进度、工单处理、评价管理等**
3. **收紧了平台服务类的L2边界，只保留真正的服务对象品类**
4. **明确了阶段一以L1+L2为主，L3仅对高频明确品类做试点**

---

## 1. 设计原则

### 1.1 核心原则

1. **以消费对象为中心：** 分类依据是"投诉涉及的是什么商品/服务"，不是投诉原因、平台、或商家类型
2. **红线：投诉问题词不进入品类树。** "退款、质量差、虚假宣传、物流慢、客服态度差、发货慢、订单问题、未履行承诺"等是投诉问题维度，不属于商品/服务品类
3. **分阶段建设：** 阶段一做L1+L2，阶段二扩展L3，阶段三完善L3并建立版本管理
4. **可扩展：** 支持未来新增L1/L2/L3品类
5. **兼容实物与服务：** 同时容纳实物商品、虚拟商品、本地生活服务、平台服务
6. **电商场景优先，兼容线下消费服务场景**

### 1.2 服务对象类型（product_service_type）

| 类型代码 | 类型名称 | 说明 | 示例 |
|---------|---------|------|------|
| PHYSICAL | 实物商品 | 有形的、需要物流交付的实体商品 | 手机、衣服、食品、家电 |
| VIRTUAL | 虚拟商品与数字服务 | 无实物形态、通过数字方式交付 | 会员、点卡、软件、在线课程 |
| LOCAL | 本地生活服务 | 需要在线下实体场所消费的服务 | 餐饮、美容、家政、酒店 |
| PLATFORM | 平台服务与账号 | 电商平台自身提供的服务 | 账号服务、店铺服务、直播服务、物流服务 |
| OTHER | 其他服务 | 不归属于以上四类的服务 | 金融服务、保险 |

### 1.3 与投诉问题维度的严格分离

**本项目主攻维度：商品/服务品类**
- 回答："投诉的对象是什么？"

**辅助维度（可后续建设，不作为品类树内容）：投诉问题类型**
- 回答："投诉的原因是什么？" → 质量问题、虚假宣传、退款纠纷、物流慢等

详见 `data_schema.md` 中的双维度数据结构设计。

## 3. 平台服务L1的边界说明（重点）

### 3.1 平台服务L1只保留"作为投诉对象的服务"

**以下内容属于平台服务品类（保留）：**

| L2 | 说明 | 示例投诉 |
|----|------|---------|
| 账号服务 | 投诉对象是平台账号本身 | "账号被无故封禁"、"无法登录" |
| 店铺与商家服务 | 投诉对象是商家后台/保证金/入驻等 | "保证金被扣"、"店铺被关" |
| 直播与内容服务 | 投诉对象是直播间/短视频功能 | "直播间被限流"、"视频无法发布" |
| 物流配送服务 | 投诉对象是快递/物流服务本身 | "快递丢失"、"物流一直不更新" |
| 平台服务其他 | 兜底 | — |

### 3.2 以下内容不属于任何商品/服务品类（已从品类树中删除）

这些是**投诉问题或处理环节**，应作为辅助维度标签：

- ~~退款纠纷~~ → 这是投诉问题，不是品类
- ~~客服响应/售后进度/工单处理~~ → 这是处理环节，不是品类
- ~~订单问题/支付异常~~ → 这是投诉问题，不是品类
- ~~评价管理/内容审核/社区纠纷~~ → 这是投诉问题，不是品类
- ~~价格保护~~ → 这是权益问题，不是品类
- ~~物流查询/签收纠纷~~ → 这是处理环节，不是品类

**如果投诉核心是关于"退款"、"客服态度"等，而完全没有商品/服务实体信息，该投诉应归入"其他/待人工复核 > 暂无法归类"。**

## 4. 阶段二品类体系（目标版本）

**启动条件：** 阶段一积累≥3000条人工校验的标注数据
**新增内容：**

1. **高频L2下拆分L3：**
   - 在标注数据≥100条的L2下，按实际投诉商品分布拆分L3
   - 每个新L3需要≥20条标注样本才能加入正式分类体系
   - 预计新增50-80个L3

2. **L3定义标准化：**
   - 每个L3需要明确：名称、典型商品词、排除条件
   - 编写L3分类指南供标注员参考

3. **低频L2保持不拆：**
   - 标注数据<50条的L2不拆L3
   - 直接以L2作为最细分类粒度

**阶段二预期规模：** 20个业务L1 + 1个兜底L1，114个L2，~80-120个试点L3

---

## 5. 阶段三品类体系（成熟版本）

**启动条件：** 阶段二运行≥3个月，积累≥10000条标注数据
**新增内容：**

1. **较完整的L3覆盖：**
   - 标注数据≥50条的L2全部拆出L3
   - 预计L3总数200-300个

2. **品类版本管理：**
   - 品类合并/拆分/废弃的正式流程
   - 历史数据回溯映射

3. **新品类自动发现：**
   - 基于新品类候选标记的统计分析
   - 自动推荐潜在新品类

**注意：** 即使是成熟版本，也不追求600个L3的全面覆盖。品类粒度应始终以"分流有意义"为准绳，而不是追求尽可能细。

## 6. 品类设计中的"禁止清单"

以下词汇/概念**禁止作为L1/L2/L3品类名称**（它们属于投诉问题维度）：

| 禁止词汇 | 所属正确维度 |
|---------|------------|
| 退款、退款纠纷、不予退款、退货不退款 | 投诉问题 |
| 虚假宣传、虚假发货、货不对板 | 投诉问题 |
| 质量问题、商品破损、商品质量问题 | 投诉问题 |
| 客服态度差、客服不处理、响应时间长 | 投诉问题 |
| 发货慢、逾期不发货、超时不到货 | 投诉问题 |
| 未履行承诺、未履行7天无理由 | 投诉问题 |
| 订单问题、订单自动取消 | 投诉问题 |

| 售后进度、工单处理 | 处理环节 |
| 投诉渠道、维权 | 处理环节 |

## 7. 品类可扩展性

### 7.1 新增L2品类
- 当某L1下出现≥3个同类投诉无法归入现有L2时，运营人员可提议新增L2
- 需要品类管理员确认
- 新增L2需提供：name、examples、所属L1

### 7.2 新增L3品类（阶段二开始）
- 当某L2下出现≥5条同类投诉无法归入现有L3时
- 新L3需要≥20条标注样本验证稳定性后才能正式加入分类体系
- 正式加入前标记为"L3候选/试用"

### 7.3 新增L1品类
- 当出现全新消费领域时（如"元宇宙虚拟商品"、"太空旅游"等）
- 需要产品经理+业务负责人+数据团队共同评审

### 7.4 品类废弃
- 连续3个月无投诉的L3可标记为"停用"
- 但保留在数据库中用于历史数据查询
- L2和L1不轻易废弃
