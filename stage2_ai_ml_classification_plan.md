## 1. 当前阶段目标

### 1.1 核心目标

**完成商品/服务品类分类的 AI/机器学习主体能力建设，稳定 L1/L2 分类结果。**

具体包括：
1. 修正当前 500 条试标数据中的明显 L1/L2 分类错误（已完成 S782/S939 修正）
2. 设计并实现规则 baseline（词典匹配 + 关键词规则）
3. 设计并实现 LLM baseline（Prompt 驱动分类）
4. 设计传统 ML baseline（TF-IDF + 分类器 / MacBERT 微调）
5. 搭建混合分类方案（规则 → ML → LLM → 复核）
6. 准备后续模型训练所需的数据结构、特征、标签和流程
7. 暂不扩展人工复核/人工验收/准确率评估流程

### 1.2 为什么暂停人工复核流程设计

- **AI/ML 主体能力未稳定前，设计人工复核流程是本末倒置。** 如果自动分类的 L1/L2 错误率还很高，人工复核会成为瓶颈。
- **当前 500 条试标数据量不足以支撑完整的人工复核流程设计。** 需要先扩展到 1000-3000 条，才能准确估计复核量级、复核难度和复核成本。
- **Phase 1 的 10 条人工复核触发规则（HR-1 至 HR-10）已足够当前使用。** `need_human_review` 字段已存在，不需要进一步细化。
- **人工准确率评估（80 条 manual 填写）暂缓。** 等 AI/ML 分类主体稳定后，再做系统化的人工评估更有效率。

### 1.3 第一阶段与第二阶段的关系


## 2. AI/ML 分类系统总体架构

### 2.1 多层漏斗架构

```
投诉文本进入
    │
    ▼
┌──────────────────────────────────────────────────┐
│ Step 0: 文本预处理                                │
│ - title + content_summary 拼接                    │
│ - 基本清洗（去重空格、标准化标点）                  │
│ - 输出: clean_text                                │
└──────────────────────┬───────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────┐
│ Step 1: 实体抽取（Entity Extraction）              │
│ - 方法: 词典匹配 + LLM NER                        │
│ - 输出: main_entity, main_entity_type,             │
│         secondary_entities, evidence_text          │
│ - 如果 extraction FAILED → 直接进入 LLM 兜底      │
└──────────────────────┬───────────────────────────┘
                       │ extraction SUCCESS
                       ▼
┌──────────────────────────────────────────────────┐
│ Layer 0: 规则/词典精确匹配（Rule Baseline）        │
│ - 商品词典精确匹配 → L1/L2 直接映射               │
│ - 品牌 + 型号规则 → L1/L2 推断                    │
│ - 高置信度关键词模式 → 分类                        │
│ - 目标: 覆盖 30-40% 样本，准确率 ≥ 95%            │
└──────────┬───────────────┬───────────────────────┘
           │ 命中 + 高置信度  │ 未命中 / 低置信度
           ▼                 ▼
    ┌──────────┐    ┌──────────────────────────────┐
    │ 直接输出  │    │ Layer 1: 轻量 ML 模型          │
    └──────────┘    │ - 目标: 覆盖 40-50% 样本       │
                    │ - L1 准确率 ≥ 85%              │
                    │ - 置信度 ≥ 0.75 则直接输出      │
                    └──────────┬───────────────────┘
                               │ 置信度 < 0.75
                               ▼
                    ┌──────────────────────────────┐
                    │ Layer 2: LLM 兜底              │
                    │ - 处理: 长尾品类 / 复杂语义     │
                    │ - 目标: 覆盖 10-20% 样本        │
                    │ - 置信度 ≥ 0.80 则输出          │
                    └──────────┬───────────────────┘
                               │ 置信度 < 0.80
                               ▼
                    ┌──────────────────────────────┐
                    │ Layer 3: 标记 need_human_review │
                    │ - 目标: ≤ 15% 样本              │
                    │ - 不展开人工复核流程设计         │
                    └──────────────────────────────┘
```

### 2.2 各层分工

| 层级 | 方法 | 覆盖目标 | 准确率目标 | 延迟 | 成本/千条 |
|------|------|---------|----------|------|----------|
| Layer 0 | 词典 + 规则 | 30-40% | ≥95% L1, ≥90% L2 | <10ms | ¥0 |
| Layer 1 | MacBERT 微调 | 40-50% | ≥85% L1, ≥75% L2 | ~50ms | ~¥1 GPU |
| Layer 2 | LLM（可替换的高性能模型） | 10-20% | ≥85% L1, ≥75% L2 | ~2s | ~¥20-30 |
| Layer 3 | 标记复核 | ≤15% | — | — | — |

**最终目标自动覆盖率（Layer 0 + Layer 1）：≥ 70%，LLM 调用控制在 ≤ 20%，人工复核 ≤ 15%。**

## 3. 输入字段设计

### 3.1 原始输入字段

从投诉数据中提取以下字段作为模型输入：

| 字段 | 类型 | 必填 | 说明 | 来源 |
|------|------|------|------|------|
| `complaint_id` | string | ✅ | 投诉唯一标识 | 原始数据 |
| `title` | string | ✅ | 投诉标题（最高信号密度） | 原始数据 |
| `content` / `content_summary` | string | ✅ | 投诉正文 / 摘要 | 原始数据 |
| `platform` | string | ✅ | 电商平台名称 | 原始数据 |
| `merchant` | string | 可选 | 商家名称（含品类线索） | 原始数据 |

### 3.2 衍生输入字段（由前置模块生成）

| 字段 | 类型 | 说明 | 生成方式 |
|------|------|------|---------|
| `clean_text` | string | 拼接清洗后的文本 = `title + [SEP] + content_summary` | 预处理 |
| `detected_product_entities` | json | 实体抽取结果（main_entity, type, secondary_entities） | 词典 + LLM NER |
| `brand_info_used` | boolean | 是否使用了品牌辅助信息进行品类推断 | 品牌模块 |
| `evidence_text` | string | 支持分类判断的原文片段 | LLM / 规则 |

### 3.3 输入文本拼接策略

```
clean_text = f"{title} [SEP] {content_summary}"
```

- 标题权重最高 → 放在最前面
- `[SEP]` 作为分隔符（MacBERT 原生支持）
- 如果 content_summary 为空，仅使用 title
- 最大长度限制：512 tokens（MacBERT）/ 4096 tokens（LLM）

### 3.4 特征工程（供 ML baseline 使用）

除了原始文本，额外构建以下特征：

| 特征 | 类型 | 说明 |
|------|------|------|
| `title_length` | int | 标题字符数 |
| `content_length` | int | 正文字符数 |
| `has_product_word` | bool | 是否包含已知商品词（来自商品词典） |
| `has_brand_word` | bool | 是否包含已知品牌词（来自品牌词典） |
| `has_service_word` | bool | 是否包含服务关键词 |
| `has_amount` | bool | 是否包含金额信息 |
| `platform_id` | categorical | 平台编码（one-hot） |
| `main_entity_type` | categorical | 主实体类型（PRODUCT/BRAND/SERVICE/PLATFORM_SERVICE） |
| `matched_product_category_l1` | string | 词典匹配到的 L1（如命中） |
| `matched_product_category_l2` | string | 词典匹配到的 L2（如命中） |
| `num_entities` | int | 抽取到的实体数量 |

## 4. 标签字段设计

### 4.1 主标签（必须标注）

| 字段 | 类型 | 说明 | 可选值 / 示例 |
|------|------|------|-------------|
| `l1_code` | string | 一级品类编码 | "01"–"20", "98"（21 类） |
| `l1_name` | string | 一级品类名称 | "食品生鲜"、"数码电子" 等 |
| `l2_code` | string | 二级品类编码 | "01-01"–"20-99"（~105 类） |
| `l2_name` | string | 二级品类名称 | "休闲零食"、"手机通讯" 等 |
| `main_entity` | string | 主商品/服务实体文本 | "显卡"、"会员"、"半日达配送" |
| `main_entity_type` | string | 主实体类型 | PRODUCT / SERVICE / PLATFORM_SERVICE / BRAND / UNKNOWN |

### 4.2 辅助标签（可选，但建议标注）

| 字段 | 类型 | 说明 |
|------|------|------|
| `product_service_type` | string | PHYSICAL / VIRTUAL / LOCAL / PLATFORM / OTHER |
| `secondary_entities` | json array | 次要实体列表 |
| `brand_main` | string | 主品牌名称（如有） |
| `complaint_issue_type` | string | 投诉问题类型（辅助维度，不用于 ML 训练） |

### 4.3 标签层级约束

**L2 必须属于 L1。** 这是一个严格的层级约束，训练和推理时都需要保证：

- 训练时：L2 标签的编码前两位必须等于 L1 编码（如 L2 "06-01" 的前两位 "06" = L1 "06"）
- 推理时：采用分层预测策略——先预测 L1（21 类），然后在 L1 对应的 L2 候选集中预测 L2
- 标签校验逻辑：
  ```
  IF left(l2_code, 2) != l1_code:
      → 标签错误，需要人工修正
  ```

### 4.4 弱标签策略（Weak Labeling）

当前 500 条试标数据在 v2.4 规则分歧裁决回写后，332 条为 auto_pass（系统高置信度），168 条为 need_human_review。这些数据可以作为**弱标签**使用：

| 标签质量 | 数量 | 用途 |
|---------|------|------|
| Auto-pass（系统高置信度，≥0.75） | 332 | 可作为弱标签训练 ML baseline |
| Need human review | 168 | 当前暂不用于 ML 弱训练，可用于后续难例分析 |
| 人工修正后的标签 | 待积累 | 金标准标签，用于评估和训练 |

**弱标签使用原则：**
- 训练 ML baseline 时可以用 auto_pass 的弱标签；当前阶段先做离线弱标签基线，不扩展人工复核流程
- 弱标签训练出的模型准确率上界受限于 LLM 分类准确率和规则回写质量
- 后续需要至少 500-1000 条人工修正标签才能训练出稳定模型

## 5. 规则 Baseline（Layer 0）

### 5.1 设计原则

- **高精确率优先于高召回率。** 规则命中的样本必须几乎 100% 正确，错误命中会削弱整个管道的信心。
- **规则是"精确打击"，不是"地毯轰炸"。** 只覆盖确定性高的模式，不确定的交给 ML 或 LLM。
- **规则可解释、可维护、可审计。** 每条规则都能追溯到具体的文本证据。

### 5.2 规则分类体系

#### A 类规则：商品词典精确匹配（最高优先级）

从投诉文本中匹配已知商品词，直接映射到 L2 品类。

**词典结构：**
```json
{
  "term": "显卡",
  "aliases": ["图形卡", "GPU", "独显", "核显", "显示适配器"],
  "l1_code": "06",
  "l1_name": "数码电子",
  "l2_code": "06-02",
  "l2_name": "电脑整机与硬件",
  "type": "PRODUCT",
  "priority": 1,
  "negative_context_keywords": ["集成显卡", "核显", "亮机卡"],
  "negative_context_note": "低端集显投诉可能不属于独立显卡品类"
}
```

**当前词典规模（待建设）：**
- 阶段一（当前）：从 500 条试标 + 高频投诉词中提取，预计 300-500 个商品词
- 阶段二（扩展）：从 3000+ 条标注数据中提取，预计 800-1500 个商品词

**精确匹配规则：**
1. 完全匹配：商品词在文本中作为独立词出现
2. 别名匹配：支持同义词、简称、俗名
3. 否定上下文过滤：如果商品词在否定上下文中出现，不触发匹配（如"不是显卡的问题"）

#### B 类规则：品牌 + 型号/产品线匹配

当文本中只有品牌和型号，没有显式商品词时，通过品牌-型号-品类映射推断。

**规则示例：**
```
IF text contains "RTX 5090" OR "5090显卡" OR "5090":
    → main_entity = "显卡"
    → L1 = "06", L2 = "06-02"
    → confidence = 0.90
    → evidence = "型号5090→NVIDIA显卡"

IF text contains "iPhone" AND ("16" OR "15" OR "14" OR "Pro" OR "Max"):
    → main_entity = "手机"
    → L1 = "06", L2 = "06-01"
    → confidence = 0.90
```

**品牌-型号映射库规模：** ≥50 条高频型号映射（手机、显卡、家电为主）

#### C 类规则：高置信度关键词模式

基于固定模式的分类规则：

| 模式 | 示例 | L1 | L2 |
|------|------|----|----|
| "会员" + 无法/过期/自动续费 | "会员无法使用" | 15 虚拟商品 | 15-01 会员与订阅服务 |
| "外卖" + 送达/迟到/少送 | "外卖送了一个小时还没到" | 16 本地生活服务 | 16-01 餐饮外卖 |
| "运费险" + 不赔/拒赔/理赔 | "运费险不赔付" | 19 金融支付与保险 | 19-02 保险产品 |
| "账号" + 封禁/冻结/无法登录 | "账号无故被封禁" | 20 平台服务 | 20-01 账号服务 |
| "保证金" + 不退/扣除 | "店铺保证金被扣" | 20 平台服务 | 20-02 店铺与商家服务 |
| "快递/物流" + 丢失/不更新/延误 | "快递丢失" | 20 平台服务 | 20-04 物流配送服务 |
| "充值" + 不到账/未到账 | "话费充值不到账" | 15 虚拟商品 | 15-04 数字卡券与充值 |

#### D 类规则：否定/排除规则（Negative Rules）

防止将投诉问题词、量词、渠道词误分类为商品品类：

| 规则 | 说明 |
|------|------|
| "退款/退货/虚假宣传/客服态度/发货慢" 等投诉问题词 | 不触发任何商品品类分类 |
| "单笔/多笔/这笔/那笔" 中的 "笔" | 量词，非文具 |
| "一包/两包/三包" 中的 "包" + 非箱包上下文 | 量词，非箱包 |
| "包裹/包装/包邮/包庇/包括/红包" | 不触发箱包皮具分类 |
| "天猫/天猫超市/猫腻" | 不触发宠物猫分类 |
| "成语中的动物词（挂羊头卖狗肉/狐假虎威等）" | 不触发宠物分类 |
| 平台名（淘宝/京东/拼多多/抖音） | 不触发商品品类（除非明确是平台服务投诉） |
| 客服名（阿里客服/XX客服） | 不触发商品品类 |
| 中文量词（件/条/双/台/瓶/盒/袋/桶） | 不作为商品实体 |

### 5.3 规则引擎输出格式

```json
{
  "rule_layer": {
    "matched": true,
    "rule_id": "DICT-GPU-001",
    "rule_type": "DICTIONARY_EXACT",
    "main_entity": "显卡",
    "l1_code": "06",
    "l1_name": "数码电子",
    "l2_code": "06-02",
    "l2_name": "电脑整机与硬件",

    "evidence_text": "5090显卡[标题]; 显卡温度[内容]",
    "matched_term": "显卡"
  }
}
```

| L1 品类 | auto_pass 数 | 预估规则可覆盖 |
|---------|-------------|-------------|

## 6. LLM Baseline（Layer 2）

### 6.1 定位

LLM 在混合方案中承担两个角色：

1. **冷启动标注引擎**（阶段一）：用 LLM 为无标签数据生成弱标签，积累训练数据
2. **兜底分类器**（Layer 2）：处理规则和 ML 模型无法覆盖的长尾/复杂 case

### 6.2 Prompt 设计

采用 **联合 Prompt**（实体抽取 + 品类分类一次调用完成）：

```
System: 你是一个电商投诉商品/服务品类分类专家。你的任务是从投诉文本中识别投诉涉及的商品或服务，并将其归类到给定的品类体系中。

## 品类体系
[动态注入 stage1_taxonomy.json 中的 L1/L2 列表，含 examples]

## 分类规则
1. 品类回答"投诉对象是什么商品/服务"，不是"为什么投诉"
2. 以下内容不是品类，不能输出为分类结果：退款、退货、质量问题、虚假宣传、物流慢、客服态度差、发货慢
3. 如果标题和正文都找不到商品/服务信息，main_entity 设为 "N/A"，分类归入 98-01
4. 如果商品/服务在现有品类体系中找不到匹配，分类归入 98-02，标记 taxonomy_gap=true
5. 多商品时，以标题首次提及、金额关联、问题关联的优先级确定主商品
6. 店铺名中的"XX旗舰店"可作为辅助线索，但不能替代明确的商品词
7. 量词（单笔订单的"笔"、一包两包的"包"）不是商品实体
8. 成语中的动物词（挂羊头卖狗肉等）不是商品实体
9. L2 必须属于 L1，编码前两位必须一致

User: ## 投诉信息
标题: {title}
内容摘要: {content_summary}
平台: {platform}
商家: {merchant}

请输出 JSON：
{
  "main_entity": "商品/服务实体文本（找不到则为 N/A）",
  "main_entity_type": "PRODUCT|SERVICE|PLATFORM_SERVICE|BRAND|N/A",
  "l1_code": "两位数字",
  "l1_name": "L1 名称",
  "l2_code": "XX-XX",
  "l2_name": "L2 名称",
  "confidence": 0.0-1.0,
  "evidence_text": "从原文中摘录的支持证据",
  "classification_reason": "一句话说明分类依据",
  "taxonomy_gap": true/false,
  "secondary_entities": []
}
```

### 6.3 模型选择

LLM 层使用**可替换的高性能大语言模型**，具体选型按以下因素评估后确定：
- 分类准确率（在标注验证集上评估 L1/L2 accuracy）
- API 调用成本（每千条）
- 可用性与调用限制（QPS、日配额）
- 输出格式稳定性（JSON 结构化输出能力）

推荐按角色分层使用不同规格的模型：

| 角色 | 适用场景 | 推荐模型规格 |
|------|---------|------------|
| 最终兜底分类器 | 高难度 case、规则和 ML 均无法覆盖 | 高性能模型（如 DeepSeek / OpenAI / Claude 旗舰系列） |
| 批量弱标签标注 | 大批量生成训练数据 | 性价比模型（中等规格，成本优先） |
| 简单规则确认 | 规则命中后的快速复核 | 轻量模型（低延迟、极低成本） |

**推荐策略：高规格模型做最终兜底和难 case，中等规格模型做大批量弱标签标注，轻量模型做简单规则确认。具体模型名称按实际评估结果确定，不在此文档中写死。**

## 7. 传统 ML Baseline（Layer 1）

### 7.1 候选方案

| 方案 | 模型 | 优点 | 缺点 | 推荐度 |
|------|------|------|------|--------|
| A | TF-IDF + Logistic Regression | 极快、可解释、无需 GPU | 效果上限低、不考虑语序 | ⭐⭐⭐ baseline only |
| B | TF-IDF + XGBoost/LightGBM | 快、处理特征交互好 | 文本理解弱 | ⭐⭐⭐ baseline only |
| C | MacBERT-base 微调 | 中文理解好、效果上限高 | 需要 GPU、训练时间较长 | ⭐⭐⭐⭐⭐ 主力 |
| D | text2vec-large-chinese + 分类器 | 语义表示好、可复用 | 模型较大 | ⭐⭐⭐⭐ 备选 |
| E | 多任务学习（实体 + 分类联合） | 共享表示、端到端 | 复杂度高、需要更多数据 | ⭐⭐⭐ 阶段三考虑 |

**推荐主力方案：C（MacBERT-base 微调），A 作为快速 baseline 对照。**

### 7.2 MacBERT 微调方案设计

#### 模型架构

```
Input: [CLS] title [SEP] content_summary [SEP]
         │
         ▼
    MacBERT-base (12 layers, 768 hidden)
         │
         ▼
    [CLS] hidden state (768-d)
         │
         ├──────────────────┐
         ▼                  ▼
    L1 Classifier      L2 Classifier
    (Linear 768→21)    (Linear 768+21→~105)
    Softmax            Constrained Softmax
                        (仅预测 L1 对应的 L2)
```

**关键设计：**
- L1 分类头：21 类（20 业务 L1 + 1 兜底 L1 "98"）
- L2 分类头：分层约束预测。先得到 L1 预测结果，然后在 L1 对应的 L2 子集中做 softmax
- 联合损失：`Loss = L1_loss + λ × L2_loss`（λ 建议 0.5-1.0）
- L1 预测错误时，L2 损失不计入（或赋低权重）——防止 L1 错的梯度污染 L2 分类头

#### 训练配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 预训练模型 | `hfl/chinese-macbert-base` | MacBERT 是成熟、开源、可控的中文预训练模型 baseline，可作为第一版中文文本分类微调方案 |
| 最大序列长度 | 512 | 覆盖大部分投诉文本 |
| Batch size | 16-32 | 取决于 GPU 显存 |
| Learning rate | 2e-5 | 标准 BERT 微调学习率 |
| Epochs | 3-5 | 配合 early stopping |
| Optimizer | AdamW | weight decay 0.01 |
| Warmup | 10% steps | 线性 warmup |
| 损失函数 | CrossEntropyLoss | L1: 标准 CE；L2: 分层 CE |
| 类别不平衡处理 | Focal Loss (γ=2) | 或 Class-weighted CE |

#### 训练数据需求

| 数据量 | L1 预期准确率 | L2 预期准确率 | 可行性 |
|--------|-------------|-------------|--------|
| 500 条（弱标签） | ~70-75% | ~60-65% | 可用于快速验证，不宜上线 |
| 1000 条（弱标签 + 部分人工修正） | ~75-80% | ~65-70% | 勉强可用 |
| 3000 条（≥1000 人工修正） | ~82-87% | ~72-78% | 推荐目标 |
| 5000+ 条（≥2000 人工修正） | ~85-90% | ~75-82% | 理想状态 |

**当前状态：500 条弱标签可用，ML 训练可做快速验证但准确率有限。**

### 7.3 数据增强策略

在标注数据有限的情况下：

1. **回译增强（Back Translation）：** 中文 → 英文 → 中文，生成变体文本
2. **同义词替换：** 商品词替换为别名（"显卡" ↔ "图形卡" ↔ "GPU"）
3. **上下文窗口滑动：** 从长文本中截取不同位置的片段
4. **LLM 改写：** 用 LLM 改写投诉文本，保持品类标签不变
5. **跨平台改写：** 将"在淘宝买了XX"改写为"在京东买了XX"

**注意：** 数据增强不改变标签，只改变输入文本表达方式。增强数据仅用于 ML 训练，不用于评估。

### 7.4 传统 ML Baseline（TF-IDF + LR/XGBoost）

用途：快速 baselines，验证数据可分性，设定"及格线"。

**特征：**
- TF-IDF unigram + bigram（top 5000 features）
- 文本统计特征（长度、实体数、平台 one-hot）
- 词典匹配特征（是否命中已知商品词/品牌词）

**预期表现：**
- L1 accuracy: ~65-70%（可作为 ML baselines 的下界参考）
- 如果 MacBERT 不能显著超越此基线 → 说明数据量不够或标签质量问题

## 8. 混合分类方案

### 8.1 推理流程（Inference Pipeline）

```
输入: complaint_id, title, content_summary, platform, merchant
    │
    ▼
[文本预处理]
    │
    ▼
[实体抽取] ──FAILED──→ [LLM 兜底] ──低置信度──→ [标记 need_human_review]
    │ SUCCESS                                        │ 高置信度
    ▼                                                ▼
[Layer 0: 规则引擎]                               [输出]
    │
    ├── 命中 + 高置信度(≥0.90) ──→ [直接输出] ✅
    │
    └── 未命中 / 低置信度
            │
            ▼
        [Layer 1: MacBERT 模型]
            │
            ├── 置信度 ≥ 0.75 ──→ [直接输出] ✅
            │
            └── 置信度 < 0.75
                    │
                    ▼
                [Layer 2: LLM 兜底]
                    │
                    ├── 置信度 ≥ 0.80 ──→ [输出] ✅
                    │
                    └── 置信度 < 0.80 ──→ [标记 need_human_review] ⚠️
```

### 8.2 置信度校准

不同层级输出的置信度需要统一校准，确保跨层可比：

| 层级 | 原始置信度 | 校准后输出 | 说明 |
|------|----------|----------|------|
| Layer 0 规则 | 0.90-0.98 | 直接使用 | 规则命中精确率极高 |
| Layer 1 ML | softmax 概率 | Platt Scaling / Isotonic Regression 校准 | ML 模型的 softmax 概率往往过于自信 |
| Layer 2 LLM | 0.0-1.0 | 使用 LLM 自评 + 规则校验 | LLM 自评置信度需要交叉验证 |

**校准方法：**
- 收集各层的预测结果 + 真实标签
- 对 Layer 1 的输出概率做 Isotonic Regression 校准
- 校准后在验证集上选择最优阈值（平衡覆盖率和准确率）

所有层级输出统一为以下格式：

### 9.2 数据划分策略

对于 ML 训练，按以下比例划分：

| 数据集 | 比例 | 用途 | 说明 |
|--------|------|------|------|
| 训练集 (train) | 70% | 模型训练 | 可使用弱标签 |
| 验证集 (val) | 15% | 超参调优、阈值选择 | 必须使用人工修正标签 |
| 测试集 (test) | 15% | 最终评估 | 必须使用人工修正标签，且不参与任何训练 |

**分层划分原则：**
- 按 L1 品类分层，确保每个 L1 在训练/验证/测试集中都有代表
- 验证集和测试集优先使用人工修正过的金标准标签
- 测试集严格隔离，只在最终评估时使用一次

### 9.3 标注优先级

数据有限时，按以下优先级分配人工修正资源：

| 优先级 | 样本类型 | 说明 |
|--------|---------|------|
| P0 | 规则引擎覆盖但可能存在边界错误的样本 | 确保规则 base 可靠 |
| P1 | 高频 L1/L2 品类的低置信度样本 | 提升高频品类准确率 |
| P2 | 多实体/歧义/复杂样本 | 补充模型对困难 case 的理解 |
| P3 | 长尾品类样本 | 确保长尾不被忽视 |
| P4 | auto_pass 高置信度样本（抽样验证） | 确认系统高置信度判断是否正确 |

## 10. 评估方式（离线验证）

### 10.1 评估原则

- **只做离线验证，不涉及人工复核流程设计。**
- **评估以 L1/L2 准确率为核心指标。**
- **分品类、分难度、分模型来源做细粒度分析。**
- **`need_human_review` 字段保留但不扩展其流程。**

### 10.2 核心评估指标

| 指标 | 定义 | 计算方式 | 当前目标 |
|------|------|---------|---------|
| L1 Accuracy | L1 预测正确的样本比例 | `correct_l1 / total` | ≥80% |
| L2 Accuracy | L2 预测正确的样本比例（L1 必须正确） | `correct_l2 / total` | ≥70% |
| Entity Hit Rate | 至少抽到 ≥1 个商品/服务实体的样本比例 | `has_entity / total` | ≥85% |
| Auto-pass Rate | Layer 0 + Layer 1 自动输出的样本比例 | `auto_pass / total` | ≥70% |
| LLM Fallback Rate | 需要 LLM 兜底的样本比例 | `llm_fallback / total` | ≤20% |
| Review Flag Rate | 标记 need_human_review 的样本比例 | `review_flag / total` | ≤15% |
| Taxonomy Gap Rate | 标记品类缺失的样本比例 | `gap / total` | ≤5% |

### 10.3 细粒度评估维度

每个评估维度都需要独立统计：

1. **按品类维度：** 每个 L1/L2 品类的准确率（发现薄弱品类）
2. **按实体类型维度：** PRODUCT vs SERVICE vs PLATFORM_SERVICE 的准确率差异
3. **按模型来源维度：** Layer 0 / Layer 1 / Layer 2 各自的准确率（验证每层是否达到目标）
4. **按文本长度维度：** 短文本 vs 长文本的准确率差异
5. **按置信度分档：** ≥0.90 / 0.75-0.90 / <0.75 各档的准确率（验证置信度校准）
6. **按平台维度：** 不同平台的准确率差异（发现平台偏差）

### 10.4 错误分析矩阵

对每条分类错误的样本，记录以下信息：

| 字段 | 说明 |
|------|------|
| `error_sample_id` | 样本 ID |
| `error_type` | ENTITY_ERROR / L1_ERROR / L2_ERROR / BOTH_ERROR |
| `error_pattern` | 重复出现的错误模式标签 |
| `predicted_l1` / `predicted_l2` | 系统预测的分类 |
| `true_l1` / `true_l2` | 真实分类 |
| `error_root_cause` | 根因分析（实体抽取错误 / 词边界误匹配 / 品牌混淆 / 平台服务混淆 / 歧义文本 / 品类缺失） |
| `fix_suggestion` | 修复建议（加词典 / 加规则 / 加 few-shot / 需人工判断） |

**目标：从错误中持续提取可修复的模式，减少同类错误重复出现。**

### 10.5 离线评估数据集

| 数据集 | 数量 | 用途 | 状态 |
|--------|------|------|------|
| 500 条试标 (v2.3) | 500 | 快速验证、规则调试 | ✅ 已有 |

## 15. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 商品词典覆盖率不足（<30%） | 中 | 规则引擎效果差，压力转移到 ML/LLM | 用 LLM 批量从 24,713 条投诉中抽取高频商品词 |
| ML 训练数据量不足导致准确率低 | 高 | MacBERT 准确率不达标 | 先用 LLM 生成更多弱标签；先只做 L1 分类（21 类）；降低 L2 目标 |
| LLM 幻觉（如 S939）再次出现 | 中 | 产生无来源实体 | 增加输出验证步骤（evidence 原文回查）；新增幻觉检测规则 |
| 长尾品类数据极度稀疏 | 高 | 部分 L2 只有 1-2 条样本 | 长尾品类不强制 ML 覆盖，由 LLM 兜底 |
| 弱标签训练的模型上界受限于 LLM 准确率 | 高 | ML 模型无法超越 LLM | 持续积累人工修正标签；逐步减少对弱标签的依赖 |
| 品类体系需要调整（如新增中性服装 L2） | 中 | 标签体系变动导致需要重新标注 | 在 ML 训练前先稳定品类体系；如有新增，做标签迁移 |
